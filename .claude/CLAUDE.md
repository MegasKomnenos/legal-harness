# Rules

- 모든 내부 사고(thinking)를 한국어로 진행할 것.

## 법률 문서 작업 시 핵심 원칙

법률 문서 생성 작업(`/legal-harness` 스킬)에서는 다음 원칙이 적용된다. 스킬이 로드되지 않은 일반 세션에서도 이 디렉토리의 법률 관련 파일을 다룰 때는 아래 원칙을 고려한다.

- **서브에이전트 제한**: 법률 문서는 사건 전체의 사실관계와 쟁점 간 관계가 하나의 컨텍스트 안에서 일관되어야 한다. 서브에이전트는 이 맥락을 공유하지 못하므로, 탐색·읽기·생성은 메인 에이전트가 Read/Glob/Grep/Bash 도구로 직접 수행하는 것을 원칙으로 한다. **예외**: 7단계와 9단계의 Phase 2 내용 평가에 한하여 Agent 도구로 서브에이전트를 생성하여 독립 평가를 수행한다.
- **전문 전체 적재 원칙**: 법률 문서에서 판례·법령을 인용하려면 전문의 사실관계와 판시 뉘앙스를 파악해야 한다. 발췌나 부분 검색은 맥락을 놓쳐 오인용 위험이 있으므로, 전문 전체를 Read 도구로 읽어 메인 컨텍스트에 직접 적재한다.
- **판례 전문 통독 원칙**: 판례를 인용하려면 `법령_판례/판례/`에서 해당 판례 파일 전체를 읽는다. 인용 사전의 요약만으로는 사실관계와 판시 뉘앙스를 파악하기 어렵다.

## 불변 자산 보호

- `harness/` 디렉토리의 파일을 직접 수정하지 않는다. 수정이 필요하면 사용자에게 확인한다.
- `법령_판례/` 디렉토리의 법령 파일을 수정·삭제하지 않는다. 판례 추가/변환은 `harness/관리_절차.md`에 따라 수행한다.

## 작문 규칙

생성하는 법률 문서와 사용자 대상 응답에 적용한다(`harness/` 참조 문서 등 메타 문서에는 강제하지 않는다).

- em dash(—)를 사용하지 않는다. 법률 문서 산출물에서는 `validate_doc.py`가 자동 검출한다.
- 과잉 설명, 같은 말 반복, 불필요한 재진술을 하지 않는다.

## 커밋·푸시 절차

작업의 버전 관리는 다음 규범을 따른다. 이 규범은 기본 동작("사용자가 요청할 때만 커밋")에 우선한다.

- **분리 단위 커밋**: 분리 가능한 단위의 작업을 마칠 때마다 그 변경을 커밋한다. working tree에 서로 독립적인 변경이 섞여 있으면 적절한 선에서 별개 커밋으로 나누고, 더 잘게 나눌 실익이 없으면 묶어 하나로 커밋해도 무방하다.
- **종료 시 푸시**: 사용자가 요청한 전체 작업이 끝나면 그때까지의 모든 커밋을 push한다.
- **브랜치**: 이 저장소는 현재 브랜치(`master` 포함)에서 직접 작업한다. 커밋·푸시를 위해 별도 브랜치를 만들지 않는다.
- **메시지**: 기존 커밋 관행을 따라 변경의 목적을 한국어 한 줄로 요약한다. `Co-Authored-By` 등 트레일러는 붙이지 않는다.
- **하네스 변경 시 선행 조건**: `harness/` 데이터·문서를 변경한 경우 `harness/관리_절차.md` 4절의 정합성 self-test(`validate_harness.py`, exit 0)를 통과시킨 후 커밋한다.
