# 알려진 충돌 사례 지도 (열린 열거 예시)

2026-07-02 시점, 이 프로젝트(legal-case-harness)에 설치된 범용 스킬들을 전수 조사해 확인한 충돌 개연성 사례. **이 목록은 열린 열거 예시이지 판정의 정본이 아니다.** 실행 시에는 설치 현행본과 로컬 규범의 실측 대조가 정본이며, 이 지도는 4단계 대조에서 놓치기 쉬운 지점을 미리 짚는 보조 자료다. 원본과 로컬 규범 어느 쪽이든 개정되면 이 지도는 낡는다.

## 이 프로젝트의 로컬 규범 지형 요약 (당시 실측)

- 기계 강제층: `.claude/settings.json` 단일 진입점. PostToolUse(Write|Edit) hook 5종(validate_step_gate, validate_graph, validate_graph_eval, validate_doc, validate_doc_eval), PreCompact 프롬프트 hook. 주석 명시: "CLAUDE.md is advisory; the HARD guards live here."
- 절차 정본층: legal-harness(11단계 문서 생성, 서브에이전트 제한과 독립 평가 예외 2곳, 전문 전체 적재), memory-harness(커밋마다 메모리 노드, dream, 3층 그래프), casenote-fetch(외부 사이트 차단 예절), extract-documents.
- advisory층: `.claude/CLAUDE.md`(내부 사고 한국어, 서브에이전트 제한, 불변 자산 보호, 열거형 예시의 개방성 판단, 작문 규칙, 작업 착수 시 기억 읽기, 커밋 절차).
- 참조 문서층: `harness/`(관리_절차, 디렉토리_지도, data 사전 3종, style 3종, quality 2종). 문서 간 중복은 validate_harness.py가 감시.
- 기억층: `기억/` 3층 그래프. 판정 보조 판단틀은 L-0005(값싼 디폴트로의 붕괴, 표상 드리프트, β offset 삼분: 기계검증은 hook/self-test로, 규칙화는 성문화로, 판단 의존은 판단검증장치로).

## 충돌 개연성 상위 사례 (스킬명과 함께)

1. **TDD Iron Law** (superpowers:test-driven-development, 이를 배경 요구하는 writing-plans·systematic-debugging·subagent-driven-development·writing-skills): "실패하는 테스트 없이 프로덕션 코드 금지". 법률 문서 등 산문 산출물에는 실패 테스트가 성립하지 않는다(축 A). 다만 의도(산출 전 실패 조건 정의로 자기기만 차단)는 로컬의 "생성 전 기준 확정 + 독립 검증 게이트"가 담당 가능.
2. **강제 호출 규범** (superpowers:using-superpowers): "1%라도 적용 가능하면 반드시 스킬 호출, 어떤 응답·질문 전에도". 도메인 워크플로(legal-harness 11단계 등)를 코드용 스킬이 가로챌 소지(축 C). SessionStart 주입형이라 커스텀본만으로 상쇄가 어려울 수 있음.
3. **브랜치 격리 강제** (superpowers:using-git-worktrees, executing-plans, subagent-driven-development): "main/master에서 동의 없이 구현 금지", worktree 격리. 로컬은 master 직접 작업이 명문 규범(축 B).
4. **서브에이전트 대량 사용** (superpowers:subagent-driven-development 최강, dispatching-parallel-agents, requesting-code-review): 태스크마다 구현·리뷰·수정 에이전트. 로컬은 맥락 일관성을 위해 서브에이전트를 제한하고 독립 평가 2곳만 예외(축 B). 다만 "독립 리뷰로 확증편향 차단"이라는 의도는 로컬 예외 구조와 동방향.
5. **테스트 스위트 통과 게이트** (superpowers:finishing-a-development-branch): npm test 등 통과 전 진행 금지 + 머지/PR 4지선다. 무테스트·무PR 환경에서 전제 자체가 부재(축 A).
6. **검증 명령 전제** (superpowers:verification-before-completion): "검증 명령 미실행 시 완료 주장 금지". 산문 산출물에는 대응 명령이 불명확하나, 로컬에는 hook 자동 검증·self-test가 있어 접속 가능(축 A/E). 의도(신선한 증거 없는 완료 주장 금지)는 보존 가치가 높음.
7. **하드코딩 경로·커밋 컨벤션** (superpowers:brainstorming의 `docs/superpowers/specs/`, writing-plans의 `feat:` 커밋, subagent-driven-development의 `.superpowers/sdd/`): 로컬 커밋 정본은 memory-harness 절차 1(분리 단위, 같은 커밋에 메모리 노드, 한국어 한 줄, 트레일러 금지)(축 B/D).
8. **영어권 톤 규범** (superpowers:receiving-code-review의 감사·동조 표현 금지, verification-before-completion의 "Great!/Done!" 금지): 표층은 영어 특정(축 D)이나 의도(검증 없는 동조·완료 선언 차단)는 로컬 작문 규칙과 동방향.
9. **스킬 저작 메타 규범** (superpowers:writing-skills, skill-creator): 영어 description 규범, 서브에이전트 압박 테스트, fork·PR 배포, `/tmp`·`open` 등 Unix 관용구. 한국어·Windows·서브에이전트 제한 환경과 다면 충돌(축 A/B/D).
10. **karpathy-guidelines**: git·서브에이전트·브랜치 강제가 없어 이식성이 가장 높음. 다만 성공 기준을 "테스트 작성·통과"로 프레이밍하는 부분(축 A)과 영어(축 D)는 남는다. "억지 수정 금지" 판정 연습의 좋은 표본.

## 사용법

4단계 대조를 마친 뒤 이 지도와 견주어, 여기 있는 사례가 대조표에 없으면 누락 후보로 재점검한다. 반대로 이 지도에 있다는 이유만으로 충돌 판정을 내리지 않는다(원본이 개정되어 해소됐을 수 있다). 새 실측에서 이 지도에 없는 유형이 나오면 이 파일에 추가한다.
