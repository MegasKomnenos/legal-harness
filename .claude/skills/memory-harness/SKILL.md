# 작업 기억 하네스

세션 사이에 휘발되는 작업 지식을 git 커밋에 결합해 보존하고, 누적된 기억을 dream 절차로 종합해 통찰과 관례로 정제하는 하네스. 이 저장소의 커밋·푸시와 작업 기억의 **정본**이다.

이 스킬은 두 절차를 가진다. 인자 없이 호출하거나 커밋·푸시·기록 맥락이면 **절차 1(커밋-기억)**, `dream` 인자거나 반성·종합 맥락이면 **절차 2(dream)**.

## 개요: 두 종류의 글, 2층 그래프

- **커밋 메시지**는 "그 커밋을 이해하는 데 필요한 지식"이다.
- **메모리 노드**는 "그 커밋 작업과 연관되어 곧 휘발될 맥락의 결정화"다. 커밋 메시지와 역할이 다르다.

노드는 두 층의 그래프를 이룬다(bipartite 아님).

- **layer-1 메모리 노드** (`기억/layer1/M-####.md`): 작업 커밋당 하나. 가장 기초적 정보 단위.
- **layer-2 dream 노드** (`기억/layer2/D-####.md`): dream의 산출물. 종합된 통찰 + 근거 노드 목록(layer-1로 향하는 directed edge). 다른 dream 노드도 근거로 삼을 수 있어 layer-2 내부 간선도 허용된다.

## 핵심 아키텍처: 헌법 / 세칙 2층

일관성과 자기수정 가능성을 동시에 얻기 위해 두 층으로 나눈다. 둘 다 개정 가능하되, 차이는 **자율 수정 가능 여부**다.

- **핵심층 (헌법) = 이 문서**: dream이 자율로는 바꾸지 못하는 최소 계약. 변경하려면 사용자에게 제안해 승인을 받는다. 그래프가 자기수정을 거쳐도 깨지지 않게 하는 더 높은 문턱일 뿐 영구 고정이 아니다.
- **가변층 (세칙) = `기억/세칙.md`**: dream이 자율로 다시 쓰는 층. 메모리·dream 본문의 형식과 관례를 담는다. 수렴이 일어나는 곳이다.

수렴은 절차를 반복한 결과로 자연히 나타나는 것이지 따로 계량·판정하는 대상이 아니다.

## 핵심층 (이 문서가 규정하는 최소 계약)

아래 1~5는 사용자 승인 없이는 dream이 바꾸지 않는다. 승인을 받으면 개정할 수 있다.

### 1. 결합 불변식
모든 작업 커밋은 정확히 하나의 layer-1 메모리 노드를 **같은 커밋에** 포함한다. 절차 1을 따르면 이를 보장한다(현재는 절차적 준수에 의존하며, 기계적 강제는 향후 과제다). 예외는 dream 커밋뿐이다(dream 노드가 곧 그 반성이므로 자기 메모리를 면제한다).

### 2. 노드 frontmatter 골격 (고정 키)
본문이 어떻게 진화하든 그래프를 기계적으로 순회할 수 있도록, 모든 노드는 다음 키를 갖는다. 이 키들만 핵심층 소관이고, 그 아래 본문은 세칙이 지배한다.

```yaml
---
id: M-0042            # 메모리는 M-####, dream은 D-####. 일련번호, 0을 채워 4자리.
layer: 1              # 1=메모리, 2=dream
title: 한 줄 제목       # 색인 가독성용
convention_version: 1  # 이 노드가 태어난 세칙 버전 (생성 시 기록, 이후 불변)
basis: [M-0040, D-0005]  # layer-2 전용. 근거 노드 id 목록 = directed edge
---
```

- `id`는 해당 layer의 기존 최대 번호 + 1(첫 메모리는 `M-0001`, 첫 dream은 `D-0001`). 메모리의 귀속 커밋은 같은 커밋에 생성되므로 `git log --diff-filter=A -- 기억/layer1/M-0042.md`로 추적된다. 해시를 본문에 적지 않는다.
- 메모리 노드에는 `basis`가 없다(layer-1은 그래프의 잎).
- `convention_version`은 노드 생성 시점에만 기록하고, 이후 본문이 정비돼도 갱신하지 않는다(어느 세칙 regime에서 태어났는지의 궤적을 보존하기 위함).
- frontmatter는 위 고정 키로 한정한다. 다른 메타데이터는 본문 최상위 섹션에 둔다. 새 frontmatter 키를 도입하는 것은 그래프 파싱 스키마를 바꾸는 일이므로 핵심층 개정으로 보아 사용자 승인 경로를 따른다.

### 3. dream 출력의 최소 골격 (섹션만 고정, 내용은 열어 둠)
dream 노드 본문은 다음 4개 최상위 섹션을 갖는다. 섹션 안의 형식·분류는 세칙이 진화시킨다.

```
## 종합          근거 노드에서 결정화한 통찰
## 근거          basis 노드 id + 각 한 줄 이유 (frontmatter 간선의 인간가독 거울)
## 제안          열린 제안 목록
## 수렴·유연성 원장  가정한 관례 / 강화·반증 증거 / 그 결과의 이동 (서술적 기록)
```

- **제안은 열린 형식**이다. 미리 정한 분류축으로 나누지 않는다. 메모리 형식, dream 절차 자체, dream 노드 구성, 프로젝트·내부 작업 등은 제안이 다룰 수 있는 예시일 뿐 거기에 국한되지 않는다. 제안은 메타적일 수 있고, 자기 형식·절차·구성, 나아가 핵심층까지 건드릴 수 있다.
- **수렴·유연성 원장**은 측정치가 아니라 서술적 기록이다. "어떤 관례를 가정했고, 무엇이 그것을 강화/반증했으며, 그래서 어디로 움직였는가"를 남긴다. 수렴한 위치에서도 반례를 편향 없이 받아들여 방향을 새로이 할 근거가 된다.

### 4. 색인
`기억/색인.md`는 layer-1·layer-2 노드를 각 한 줄로 나열한다. layer-2 줄은 `basis`를 포함해 그래프 인접면을 노출한다. 노드 본문이 정본이고 색인은 그 거울이다.

### 5. 권한·정비 경계
- 세칙(`기억/세칙.md`)과 메모리·dream 노드는 dream이 자율로 쓸 수 있다(dream-writable). dream이 세칙을 고치면 세칙의 `convention_version`을 1 올린다.
- **메모리는 살아있는 층이다.** 프로젝트가 진행되면 메모리는 자연히 outdated해진다. dream은 현 상태와 정합하도록 노드를 첨삭·추가하고, obsolete해진 노드는 삭제한다. 다만 결정화된 보존이 기본이므로 **기존 본문의 수정은 한정적**이다. 무엇이 한정적 수정인지의 출발 기준은 `기억/세칙.md`의 "메모리 본문 수정 원칙"을 따르며 dream이 진화시킨다. 그래프 크기 관리는 이 정합 유지의 부산물이지 별도 목표가 아니다.
- **삭제 시 무결성**: 노드를 삭제할 때 그 id를 `basis`로 참조하는 dream 노드가 있으면, dangling 간선을 남기지 않도록 그 참조를 함께 정리하거나(갱신) 삭제 대신 본문에 obsolete 표시만 한다.
- **핵심층(이 문서)과 프로젝트 소스는 dream의 자율수정 대상이 아니다.** 변경하려면 사용자에게 제안해 승인을 받는다. 핵심층 개정도 이 경로로만 가능하다.

## 절차 1: 커밋-기억 (커밋 시 적용)

이 저장소의 커밋·푸시 정본이다. 기본 동작("사용자가 요청할 때만 커밋")에 우선한다.

1. **작업 단위 식별**: 분리 가능한 단위로 변경을 나눈다. working tree에 서로 독립적인 변경이 섞여 있으면 적절한 선에서 별개 커밋으로 나누고, 더 잘게 나눌 실익이 없으면 묶어 하나로 커밋한다. 이 저장소는 현재 브랜치(`master` 포함)에서 직접 작업하며 커밋·푸시를 위해 별도 브랜치를 만들지 않는다.
2. **하네스 게이트**: `harness/` 데이터·문서를 변경했으면 `python harness/scripts/validate_harness.py`를 실행해 exit 0을 확인한다(`harness/관리_절차.md` 4절). exit 1이면 불일치를 수정한 뒤 통과시킨다.
3. **메모리 노드 작성**: `기억/세칙.md`의 현행 템플릿을 읽고, 이 커밋에 대응하는 `기억/layer1/M-####.md`를 핵심층 frontmatter + 세칙 본문 형식으로 작성한다. **이 시점의 컨텍스트 윈도우에 살아있는, 이 커밋 작업과 연관된 곧 휘발될 맥락을 결정화한다**(사후 회고가 아니라 지금 살아있는 맥락의 보존이며, 작업 자체에 국한되지 않는다). 무엇을 결정화할지의 선별은 세칙의 현행 관례를 따른다. 사소한 커밋이면 짧아도 되지만 생략하지는 않는다.
4. **색인 갱신**: `기억/색인.md`의 layer-1 절에 한 줄을 추가한다.
5. **동일 커밋으로 커밋**: 작업 변경 + 메모리 노드 + 색인 갱신을 **하나의 커밋**으로 stage·commit한다. 메시지는 변경의 목적을 한국어 한 줄로 요약하고, `Co-Authored-By` 등 트레일러는 붙이지 않는다.
6. **종료 시 푸시**: 사용자가 요청한 전체 작업이 끝나면 그때까지의 모든 커밋을 push한다.

## 절차 2: dream (반성, 완전 수동 호출)

dream은 언제나 사용자의 명시 호출로만 시작한다. 자동 트리거·주기는 없다.

1. **범위**: 기본은 마지막 dream 이후 누적된 layer-1 노드 전부 + 그와 연결되는 prior layer-2 노드(이전 dream이 다룬 통찰 중 현 종합과 이어지는 것). 호출 시 범위가 지정되면 그것을 우선한다. 전문 전체를 Read로 메인 컨텍스트에 적재한다(발췌·검색은 맥락을 놓친다).
2. **종합**: 핵심층 3의 4개 골격 섹션으로 종합한다. 세부 형식은 세칙을 따른다.
3. **제안**: 열린 제안을 산출한다. 미리 정한 분류축·태그 양식을 두지 않는다.
4. **적용**: 기억 시스템 내부는 dream이 스스로 적용한다. 곧 세칙 편집(`convention_version` 증가)과, 현 프로젝트 상태에 맞춘 메모리·dream 노드의 첨삭·추가·삭제(본문 수정은 한정적). 반면 **핵심층 변경과 프로젝트 소스 변경은 적용하지 않고 dream 노드의 제안 섹션에 적어** 사용자 승인을 기다린다(파일 수정·커밋은 하지 않는다).
5. **기록·커밋**: `기억/layer2/D-####.md`를 작성하고 색인을 갱신한 뒤, dream 노드 + 내부 정비를 **dream 커밋**으로 커밋한다(자기 메모리 면제). 종료 시 push.

## 세칙 참조

본문의 구체 형식(메모리 템플릿, 결정화 선별 관례, dream 휴리스틱 등)은 모두 `기억/세칙.md`에 있다. 노드를 쓰기 전 세칙의 현행 내용을 읽는다. 세칙은 dream이 진화시키므로 시점마다 다를 수 있다.

## 기존 거버넌스 정합

- 커밋·푸시의 정본은 이 스킬이다. `.claude/CLAUDE.md`의 "커밋·푸시 절차"가 이 스킬을 가리키며, `harness/관리_절차.md` 5절과 `legal-harness` 11단계의 커밋 단계는 그 `.claude/CLAUDE.md`를 경유해 이 스킬을 따른다(직접 참조가 아니라 정본 체인을 경유).
- `harness/`·`법령_판례/`는 불변 자산이다. `기억/`는 그 밖의 신규 디렉토리이므로 보호 규칙과 충돌하지 않는다.
- Claude 표준 메모리 시스템과 사건 폴더의 `진행로그_`·`검증이력_`(per-event 작업 아티팩트)은 이 시스템과 별개이며 통합·동기화하지 않는다.
