# 관리 절차

하네스의 데이터 관리를 위한 표준 절차. 판례 추가, case 추가, 기존 case 수정의 세 가지 절차를 정의한다.

---

## 1. 판례 추가 절차

새로운 판례를 하네스에 등록하는 절차.

1. 판례 PDF를 `법령_판례/판례/`에 배치한다.
2. `python extract_all.py <PDF경로>`를 Bash로 실행한다. 이 스크립트는 추출 텍스트를 stdout으로만 출력하며 파일을 생성하지 않는다(로그는 stderr).
3. stdout 출력을 아래 표준 형식으로 정리하여 `{사건번호}.txt`로 저장한다.
4. AI 교정 절차에 따라 추출 결과를 교정한다.
5. 원본 PDF를 `법령_판례/archive/판례/`로 이동한다.
6. `harness/data/판례_인용_사전.md`에 새 판례 엔트리를 추가한다.
   - P-번호 (기존 최대 번호 + 1)
   - 정식 인용 형식
   - 법리 태그
   - 인용 원문
   - 사용처
7. 해당 법리의 `harness/data/법리_데이터베이스.md` 엔트리에 판례를 추가한다.
   - 핵심 판례 목록에 추가
   - 판시사항 원문 추가

### 표준 판례 txt 형식

```
# [법원명] [선고일] 선고 [사건번호] [재판유형]

## 판시사항
(판시사항 본문)

## 판결요지
(판결요지 본문)

## 주문
(주문 본문)

## 이유
(이유 본문)
```

- 섹션이 원문에 존재하지 않으면 해당 섹션을 생략한다.
- CaseNote 형식 PDF는 판시사항·판결요지가 별도 섹션으로 구분되어 있으므로 그대로 옮긴다.
- 일반 PDF는 본문에서 판시사항·이유 등을 식별하여 구분한다.

### AI 교정 절차

추출 스크립트(`extract_all.py`)는 Tesseract OCR과 아티팩트 자동 정리를 수행하나, 다음 유형의 오류는 기계적 정리만으로 해결되지 않는다:

- OCR 문자 치환 오류 (예: `을` → `올`, `CHS AAI` → `(다)목`)
- 표·서식의 구조 붕괴 (조문 번호·괄호 분리, 셀 내용 개별 줄 분리)
- 웹 UI 잔재 (CaseNote 캡처 PDF에서 혼입된 메뉴·버튼 텍스트)
- 원본 PDF 특성상 누락된 띄어쓰기

추출 완료 후, 메인 에이전트가 txt 파일 전체를 읽고 위 오류를 교정한다. 교정 시 다음 원칙을 준수한다:

1. **원문 표현 보존**: 원문에 있는 표현을 바꾸지 않는다. OCR이 잘못 읽은 글자를 원래 글자로 복원하는 것만 허용한다.
2. **추가 금지**: 원문에 없는 내용을 보충하거나 문장을 재구성하지 않는다.
3. **구조 복원**: 조문 번호(`제X조`)와 괄호가 분리되었으면 한 줄로 합친다. 표의 열 구분이 필요하면 탭 또는 `|` 구분자로 정리한다.
4. **잔재 제거**: CaseNote 웹 UI 텍스트(사용자명, 검색바, 메뉴, 버튼)가 남아 있으면 삭제한다.
5. **판단 유보**: 원문 표현이 확실하지 않으면 교정하지 않고 `[OCR 불확실]`로 표시한다.

## 1-1. 법령·내규·기타 PDF 변환 절차

판례가 아닌 법령, 시행령, 내규, 별표·고시, 기타 참고자료 PDF를 텍스트로 변환하는 절차.

1. PDF를 해당 하위 디렉토리(`법령_판례/법률_시행령/`, `법령_판례/경북대_내규/`, `법령_판례/별표_고시/`, `법령_판례/기타_참고/`, `법령_판례/재결례/`)에 배치한다.
2. `python extract_all.py <PDF경로> > <동일경로>.txt`를 Bash로 실행한다. 이 스크립트는 추출 텍스트를 stdout으로만 출력하므로 `>` 리다이렉트로 파일에 저장한다(로그는 stderr로 출력되어 파일에 포함되지 않는다).
3. AI 교정 절차에 따라 추출 결과를 교정한다.
4. 원본 PDF를 `법령_판례/archive/<하위디렉토리>/`로 이동한다.

### archive 디렉토리 구조

```
법령_판례/archive/
├── 판례/
├── 재결례/
├── 경북대_내규/
├── 별표_고시/
├── 기타_참고/
└── 법률_시행령/
```

원본 PDF는 삭제하지 않고 archive에 보관한다. 하네스가 읽는 파일은 txt만 사용한다.

## 2. 새 case 추가 절차

새로운 사건을 하네스에 등록하는 절차.

1. `cases/{번호}_{사건명}/` 디렉토리를 생성한다. 번호는 기존 최대 번호 + 1.
2. 사건 폴더 안에 하위 디렉토리를 생성하고 문서를 배치한다. 기존 사건(01~03)의 구조를 따른다:
   - `절차_정보공개/`, `절차_행정심판/`, `절차_민원/` (절차 문서)
   - `작성_최종/`, `작성_초안/` (작성 문서)
   - `민원/`, `증거/`, `분석/`, `외부소통/` (필요 시)
4. 관련 법령·판례를 `법령_판례/`에 배치한다. 미수록 판례는 "1. 판례 추가 절차"에 따라 등록한다.
5. `harness/디렉토리_지도.md`에 새 사건 섹션을 추가한다:
   - 사건 요약, 피청구기관, 문서 분류 표, 관련 문서 참조 가이드

---

## 3. 기존 case 수정 절차

기존 사건에 새 문서가 추가되거나 상황이 변경된 경우의 절차.

1. 새 절차 문서(답변서, 결정통지서, 보충서면 등) 수신 시 해당 `절차_*/` 디렉토리에 배치한다.
2. 기존 진행로그·검증이력은 보존한다(`archive/`에 보관되어 있을 수 있다). 덮어쓰지 않는다.
3. 추가 판례가 필요하면 "1. 판례 추가 절차"에 따라 등록한다.
4. `harness/디렉토리_지도.md`의 해당 사건 문서 분류 표에 새 문서를 추가한다.
5. 새 문서에 대해 `/legal-harness`를 실행할 때는 1단계부터 전체 절차를 수행한다.

---

## 4. 정합성 점검

판례 추가, case 추가, case 수정 등 하네스 데이터·문서를 변경한 후에는 정합성 self-test를 실행한다. 여러 문서에 중복 기재된 정보(판례 사전·법리DB·디렉토리 지도·README·SKILL)가 서로 어긋나지 않는지 자동 점검한다.

1. `python harness/scripts/validate_harness.py`를 Bash로 실행한다.
2. 출력된 불일치 항목(판례 사전 ↔ 판례 파일, 법리DB ↔ 인용사전, 법조문 원문 미보유, Phase 2 파싱, README harness 경로, extract SKILL OCR 등)을 수정한다.
3. 의도적으로 보류하는 공백(원문 미확보 판례·법령 등)은 스크립트의 `KNOWN_GAPS`에 사유와 함께 등록하여 경보에서 제외한다. 공백이 해소되면 해당 항목을 `KNOWN_GAPS`에서 제거한다.
4. exit 0(모든 점검 통과)을 확인한 후 변경을 마무리한다.
