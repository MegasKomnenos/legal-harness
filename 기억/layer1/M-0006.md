## 맥락

보충서면(2026-10986)의 10단계 PDF 생성이 이 Windows 환경에서 실패했다. `generate_pdf.py`가 WeasyPrint 전용인데, WeasyPrint는 GTK 런타임(libgobject-2.0 등 Pango·Cairo·GObject)을 native 의존으로 요구하고 이 환경엔 그것이 없다(`pip install weasyprint`로 Python 패키지는 깔려도 import 시 OSError). 사용자가 "PDF 생성을 위한 별도 절차를 고안해 하네스에 반영"하라고 지시하여, `generate_pdf.py`를 폴백 체인으로 확장하고 `harness/quality/PDF_서식.md`에 절차를 갱신했다(harness 자산, self-test exit 0 확인).

## 터득한 것

- **`generate_pdf.py`는 `parse()`(txt→구조) → `render()`(구조→HTML/CSS) → 변환의 3단이라, HTML 렌더링은 그대로 두고 변환 백엔드만 교체하면 된다.** `--html` 옵션이 이미 중간 HTML을 노출한다. 그래서 엔진 교체가 서식에 영향을 주지 않는다.
- **Playwright는 `chromium.launch(channel="msedge"/"chrome")`로 시스템 브라우저를 재사용한다.** 즉 `pip install playwright`만 하면 되고, 번들 Chromium 다운로드(`playwright install`, ~150MB)는 불필요하다. Windows엔 Edge가 기본 탑재되어 가용성이 높다.
- **Chromium은 CSS Paged Media의 margin box(`@bottom-center`)를 지원하지 않는다.** WeasyPrint는 지원한다. 그래서 페이지번호 `- N -`는 WeasyPrint 경로에서는 CSS `@bottom-center`로, Playwright 경로에서는 `page.pdf`의 `footer_template`(`<span class="pageNumber">`)으로 재현해야 양 경로가 동일 서식을 낸다. 여백도 Playwright에선 `page.pdf` margin + `prefer_css_page_size=False`로 처리.
- 검증: 7페이지 PDF에 "- 1 -" ~ "- 7 -"이 하단 중앙에 정확히 출력됨을 PyMuPDF로 확인했다.

## 작업 너머 시사점

harness의 도구(스크립트)가 특정 환경 의존성(GTK)에 묶이면 다른 환경에서 막힌다. **같은 HTML/CSS를 복수 렌더러가 공유하는 폴백 체인**은 서식 일관성을 유지하면서 환경 가용성을 넓히는 견고한 패턴이다. PDF 시각 검증 도구(pdftoppm/poppler)도 이 환경엔 없어, PyMuPDF(fitz) 텍스트 추출로 페이지·footer를 확인했다(시각 렌더 대안).
