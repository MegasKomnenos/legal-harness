## 맥락

사용자 첨삭 지시로 2026-10986 보충서면 Ⅲ.2를 재구성. 지시 3건: (1) Ⅲ.2.나의 "일부 거부한 처분"(2016두44674) 논거는 무의미하니 제15조 제1항의 원칙(전자적 보유 정보는 청구인 요청 시 원칙적으로 응)·예외(정보의 성질상 현저히 곤란) 법리를 Ⅱ에 신설하고 Ⅲ.2.나에서 그 자리를 대체, (2) Ⅲ.2.다("현저히 곤란한 경우에 해당하지 아니함")는 주장이 아니라 입증의 목적지(결론)이므로 소결로 통합하고 "현저히 곤란한 경우가 무엇인지"를 정의하지 말 것, (3) Ⅲ.3.라의 "청구인은 ~ 개연성을 입증한 것입니다"는 청구인이 능동 입증한 것이 아니므로 "개연성이 인정된다"로. legal-harness 1~11단계를 밟지 말고 일반 첨삭으로 처리하라는 지시였다. 이 커밋에는 이전 세션이 커밋하지 않고 남긴 보충서면 전반 윤문(답변서 요지 정리, Ⅲ.1 표현 다듬기 등)도 함께 포함된다(같은 파일·같은 맥락, interactive add 불가로 분리 실익 없음).

## 터득한 것 / 결정

- **2016두44674의 성격**: 이 판례는 "공개방법 제한 = 일부 거부처분 = 항고소송 대상"이라는 처분성 판례(파기환송, 본안 미판단)이지 방문열람 제한의 본안 위법성을 인정한 판례가 아니다. 제15조 제1항 조문 자체가 원칙(응의무)+예외(현저히 곤란)를 명문화하므로 판례 없이 조문만으로 본안 논증이 성립한다. 그래프가 이미 채택한 '처분성 논거 배제'(FOI-17 제외) 전략과도 정합. 7단계 서브에이전트 독립 평가도 "조문 기반 포섭으로 충분, 판례 부재가 논증을 약화시키지 않음"으로 확인. [[M-0005]]의 역원용 축(2009두19021·2003두12707)은 유지된다.
- **"제15조 제1항 단서" 오류**: 제15조 제1항은 "다만" 단서가 아니라 "그 정보의 성질상 현저히 곤란한 경우를 제외하고는"이라는 본문 내 제외 구조. "단서"는 부정확 → "제15조 제1항이 정한"으로 교정. 제9조 제1항은 "다만" 단서 구조여서 "단서 각 호"가 맞음 - 두 조문의 구조를 구별해야 한다.
- **결론형 항목의 배치**: "~에 해당하지 아니함" 류의 결론형 목차는 독립 논증항목이 아니라 소결에 둔다. 소결은 앞 논증(공개내용/공개방법 구별·2003두12707 역원용)을 받아 "그에 해당하지 않는다"는 결론만 내고 요건의 적극적 정의는 하지 않는다.
- **비상식 표현 유형**: "청구인이 개연성을 입증한 것"은 주어가 잘못됨(청구인이 계약·발주의 당사자가 아니므로 능동 입증 주체가 아님). 객관적 사실에서 개연성이 도출되므로 "보유 개연성이 인정된다"(2003두9459 판시 문언과 일치)로. [[M-0004]]의 FOI-09 입증책임 정합과 같은 맥락.
- **동기화 연쇄**: 문서에서 판례 1건을 빼면 archive 법리그래프(JSON cases·mermaid·판례표)와 harness 판례인용사전 P-015 사용처까지 동기화해야 정합. validate_graph.py는 JSON cases vs 문서 판례만 대조(mermaid·표는 무시), validate_graph_eval.py의 hash는 JSON 블록만 계산하므로 검증이력·mermaid·표 수정은 hash 불변 → 검증이력에 pass hash를 먼저 기록하면 이후 mermaid·표 Edit는 already_passed로 스킵된다. [[M-0008]]의 hook 비대칭(archive 그래프는 자동 검증, 문서는 수동) 관찰의 연장.

## 작업 너머 시사점

세션 시작 시 이미 M 상태인 파일이 있으면(이전 세션 미커밋) 커밋 전 `git diff HEAD`로 내 변경 외 잔존 변경을 반드시 파악해야 한다. 같은 파일 내 이전+현재 변경은 interactive add 불가 환경에서 분리 커밋이 어려우므로 묶어 커밋하되, 커밋 메시지·메모리에 두 변경을 모두 반영한다. harness 자산(판례인용사전 등) 수정은 사용자 승인 + `validate_harness.py` exit 0이 선행 조건이다([[M-0007]] 사용처 정비와 동종 작업).
