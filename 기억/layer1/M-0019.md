## 맥락

승인된 dream 제안 5건 중 harness 문서 관례 2건(3/3 커밋, 마지막). (3a) [[D-0003]] 제안을 `harness/style/논증_가이드라인.md` §14 "정보공개 거부처분 청구인 보충서면: 공격 표면 최소화"로 성문화. (3b) [[D-0004]] 제안을 `harness/관리_절차.md` 3-1 "사건 자산(증거·제출 문서) 추출·보관 절차"로 성문화.

## 터득한 것

- **§14는 [[D-0003]]을 가이드라인 형식으로 압축했고, 발현이 기존 절에 흩어져 있어 상위 렌즈로 묶었다.** D-0003의 5발현 중 미원용 판례(§5)·청구범위 제약(§6)·조문판례 용어(§8)·역원용(§11)은 이미 개별 절에 있었다. §14는 "공격 표면 최소화"라는 상위 렌즈로 이들을 묶고, 기존 절에 없던 것(제7호 일면구조, 계약내용성, 매크로 IRAC 전진배치)을 더했다. 발현1은 D-0003 적대 검증이 교정한 "미원용 **불리** 판례"(유리 아님)를 그대로 반영했다. 정보공개 거부·청구인·보충서면 한정임을 절 말미에 명시해 과대일반화를 막았다([[D-0003]] 제안의 "총칙 승격 대신 한정 사례" 지침 준수).
- **3-1은 관리_절차의 실제 공백을 메웠다.** 기존 3절(기존 case 수정)은 "새 절차 문서를 절차_*/에 배치"만 규정하고, 증거·제출 자산의 추출·원본 archive 이동·완전성 점검·교차 복원·메타데이터 병기는 어디에도 없었다([[M-0002]]·[[M-0003]]이 실천했으나 절차로 성문화되지 않았던 것). [[D-0004]]의 "원본이 진위의 원천, 파생물은 대조로 검증"을 5개 절차 항목으로 옮겼다.
- **두 문서 모두 [[D-0007]] 원리(정본 직접 대조)를 담는다.** §14의 "청구인은 표면을 만들지 않고 상대 표면만 해체"는 상대 답변서(정본)에 논증을 정박시키는 것이고, 3-1의 "원본이 진위 원천"은 파생물 대신 원본을 보라는 것이다. 승인된 제안들이 결국 같은 상위 원리로 수렴함을 문서화 단계에서 다시 확인했다.

## 작업 너머 시사점

- 이로써 dream 루프(D-0001~D-0008)가 낸 승인 대기 제안 5건이 3커밋(M-0017 핵심층·M-0018 검증자동화·M-0019 문서관례)으로 모두 적용됐다. dream 통찰이 제안→사용자 승인→소스 성문화로 완결되는 경로가 처음 끝까지 돌았다. [[D-0008]]이 종합한 거버넌스 2층(헌법/세칙)에서 헌법·프로젝트 소스 변경이 승인 게이트를 거쳐 반영되는 흐름의 실증이다.
- 이 3개 M 노드(M-0017~M-0019)는 dream 종료 후 쌓인 layer-1이다. 다음 dream이 열리면 "dream 제안의 적용" 자체가 하나의 주제군이 될 수 있고, 특히 이 적용들이 원 제안의 의도를 충실히 구현했는지·부작용은 없는지를 [[D-0006]] 병렬 검증이나 순차 심화로 돌아볼 후보다.
- **세칙 '커밋 직전 git status 재확인' 관례가 첫 실전 작동을 했다.** 커밋 3 직전 재확인에서, 내가 건드리지 않은 판례 파일 2건(`법령_판례/판례/대법원_2020두31408.txt`·`대법원_2011두2361_-_CaseNote.txt`)이 modified 상태임을 발견했다. diff 확인 결과 파일 첫 줄 헤더가 파일명·실제 판례와 어긋났던 것(2020두31408 파일의 헤더가 '2009두19021'로 stale)을 동시 세션/사용자가 정정 중인 것으로, [[D-0007]]의 표상(헤더) 드리프트 정정에 해당한다. 불변 자산이자 내 작업이 아니므로 커밋에서 제외하고 그대로 두었다. `git add -A`로 뭉뚱그렸다면 섞였을 변경을, 명시적 파일 지정과 사전 재확인이 갈라냈다. [[M-0016]]이 경고하고 세칙 v6에 넣은 동시 세션·OneDrive 실시간 변경 시나리오의 실증이다.
