## 맥락

[[M-0025]] 성문화 후 사용자가 "extract_all.py에 벡터 PDF 자동 판정 넣어줘"로 코드 개선을 지시. extract_all.py는 불변 관행이나 사용자 직접 지시로 수정. 먼저 코드를 정독하고 대표 PDF 분포를 진단해 무엇을 고쳐야 하는지부터 확정했다.

## 터득한 것

- **자동 판정은 이미 있었다**: `extract_pdf`는 페이지별로 `len(text)>50`이면 텍스트레이어, 아니면 이미지 OCR / `drawings>100` 벡터 렌더 OCR로 분기한다(line 271, method `vector_ocr`). 벡터 path로 글자를 그린 PDF는 `get_text()`가 0에 가까워 조건1을 통과 못하고 이미 벡터 분기로 **정확히** 라우팅됐다. 판례가 깨진 진짜 원인은 판정 부재가 아니라 그 분기의 **450dpi + psm 기본값 3**이었다(psm 3이 다단 웹캡처를 파편화).
- **요청받은 '부실 텍스트레이어 감지'(b)는 위험 실증으로 기각**: 진단 결과 경북대 학칙(정상 텍스트 PDF)이 textlen 218,270 · drawings **292,644**로, drawings가 텍스트보다 많다. drawings 수로는 벡터-글자 PDF와 정상 표-많은 PDF를 구별할 수 없어, drawings 기반 재판정은 학칙 139쪽을 벡터로 오판해 렌더 OCR하는 파괴를 부른다. 신뢰 지표는 text 길이뿐이고 이미 조건1이 그것을 쓴다. 벡터-글자는 물리적으로 get_text≈0이라 "text>50인데 벡터"라는 (b)의 전제는 거의 발생하지 않는다. **사용자 지시라도 검증 결과 위험하면 넣지 않고 설명하는 것이 정합성**([[D-0007]] 정본 대조·[[D-0001]] 값싼 디폴트 회피).
- **적용**: 조건3(벡터 분기)만 `dpi=600`, `ocr_image_file(tmp_path, psm=6)`으로. `ocr_image_file` 기본 psm=3과 전체스캔(psm4)·이미지OCR(psm3) 경로는 불변 → 정상 PDF 무영향. 회귀 테스트로 확인: 2020두31408은 파편→문장(vector_ocr), 정보공개법 PDF는 pdf_text 유지.
- SKILL(extract-documents) 판정 로직 설명도 450→600·psm6으로 동기화([[M-0016]] 문서-코드 드리프트 방지).

## 작업 너머 시사점

- 세 번째 층위의 자기개선: 발견(A-3)→판례 정정(M-0020·0023·0024)→절차 성문화(M-0025)→**코드 근본 개선(M-0026)**. 사용자 지시의 표현("자동 판정")과 실제 필요(파라미터)가 어긋날 때 코드 정본으로 확인해 조정한 것이 핵심.
- 남은 개선 여지(유보): 조건3 진입 전 전체스캔/콘텐츠이미지 판정 순서, psm을 페이지 특성별로 자동 선택하는 것 등은 실익 대비 복잡도가 커 현 단순 개선으로 갈음.
