## 맥락

앞선 세 정정 커밋([[M-0029]]·[[M-0030]]·[[M-0031]])의 메타데이터 오류를 바로잡는 커밋이다. 사용자 정합성 감사가 진행되는 동안, **동시 dream 세션이 세칙을 v9→v10으로 올렸다**([[D-0012]], 커밋 1fb5ba9, 23:47). 그 변경이 OneDrive 동기화로 working tree에 들어왔으나, 나는 감사 초반에 읽은 v9를 기억한 채 M-0029~31을 작성해 `convention_version: 9`로 잘못 기록했다(내 첫 커밋은 00:15로 세칙 상향 이후였다).

## 터득한 것

- `convention_version`은 노드 생성 시점의 세칙 버전을 적는 궤적 메타데이터다(핵심층 2). 내 세 노드는 실제로 **v10 regime에서 태어났으므로** v9는 생성 시점 오기다(이후 갱신 금지 규정과 무관한 '정확한 생성값으로의 정정'). 세칙을 재확인해 v10으로 교정했다.
- 이 오류는 [[D-0013]]("개선의 진단·처방도 표상이다 — 정본 대조 없이 드리프트한다")의 즉각적 실증이다. **정합성을 감사하는 주체(나) 자신의 노드 메타데이터가 정본(세칙)에서 드리프트했다.** [[M-0020]]의 "감사 주체도 외부 관점이 필요"가 세칙 버전 차원에서 재발했다.

## 작업 너머 시사점

- **offset 후보(다음 dream 제안감)**: 세칙 "dream 루프 운영"에 있는 "커밋 직전 git status 재확인"(동시 세션 대비) 관례를, **"메모리 노드 작성 직전 세칙 `convention_version` 재확인"**으로 확장한다. 이 저장소는 OneDrive 동기화·동시 세션으로 정본이 실시간 변한다([[M-0016]]). 커밋 직전 git status만으로는 프런트매터에 이미 박힌 낡은 버전 값을 잡지 못한다 — 노드 작성 시점에 세칙 헤더를 다시 읽어야 한다.
- **감사와 dream의 공명**: 이 감사 세션이 발견한 것(자기서술 드리프트 C-1~3, 검증 미적용 A계열)과 동시 dream이 종합한 것(D-0012 재귀적 편향, D-0013 개선도 표상)이 독립적으로 같은 뿌리([[D-0007]] 표상 드리프트, [[D-0001]] 편향)에 닿았다. 외부 관점(사용자 감사)과 내부 반성(dream)이 수렴한 실측이다.
