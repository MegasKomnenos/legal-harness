# TDD 커스텀판 생성(adapt-skill 실전 1건, 재현 우선 코드 수정)

## 맥락

adapt-skill의 실전 검증으로 `superpowers:test-driven-development`(6.1.0, SKILL.md 371줄 + testing-anti-patterns.md 299줄)를 절차 0~8단계대로 커스텀화해 `.claude/skills/test-driven-development/`를 생성했다. 6단계 게이트는 승인된 플랜의 특례(사용자 무응답 시 권고안대로 잠정 진행, 최종 보고에서 대조표+생성물 일괄 검토)로 통과시켰으므로 사용자 사후 검토가 남아 있다.

판정 결과가 커스텀판의 골격이다. 실측(테스트 러너·테스트 파일이 프로젝트 전체에 전무, legal-workbench도 npm test 미정의)을 근거로 Iron Law를 "실패하는 재현 실행 없이 수정 코드를 확정하지 않는다"로 재정식화하고(A축), 산문 산출물은 legal-harness 검증 루프 관할로 배제하는 관할 지도를 description과 적용 범위에 박았다(C축). 대조표는 지시 단위 24행(원본 유래 23 + 역방향 발견 1)이고 생성물의 `references/원본_대조.md`가 정본이다.

## 터득한 것

- 독립 반증 검증이 설계 의도대로 실질 발견을 냈고 전부 수용했다(1건 부분 수용). 가장 값진 발견은 "표층 D 판정 아래 숨은 A 전제": 원본의 합리화 반박 논증(Why Order Matters·Rationalizations 표)은 언어만 바꾸면 되는 D로 보였으나, 반박 논지 자체가 상주 테스트 스위트의 존재("can't re-run", "prevents regressions")를 논거로 쓰고 있어, 표층 변환만 하면 커스텀본이 자기 방법론(일회성 재현 실행)을 스스로 반박하는 자기모순 텍스트가 된다. 논지 재구성으로 처리했다(재실행 가능성은 "재현 입력의 기록·보존"으로 격하해 정직하게 서술).
- 역방향 점검(로컬 규범에서 대상으로)이 원본에 대응물이 없는 필수 행(T-24)을 잡았다: 이 프로젝트 코드의 다수가 `harness/scripts/`에 있는데 harness/는 불변 자산이라 수정 전 사용자 확인이 선행해야 한다. 이 게이트 없이는 커스텀본이 로컬 보호 규범을 우회하는 절차서가 된다. 지시에서 로컬로의 단방향 대조만으로는 구조적으로 잡을 수 없는 유형이다.
- 동반 리소스의 "압축 흡수"는 손실 위험이 있다. testing-anti-patterns.md를 2원리로 압축했다가 반증이 AP3(불확실하면 실물 먼저 실행 후 관찰)·AP4(축소 입력은 실물의 구조 특성 보존)의 손실을 잡아 4원리로 확장했다. AP4는 로컬 특이성과 정확히 맞물린다: extract_all.py의 페이지 분기가 입력의 구조 특성(textlen·drawings·픽셀)에 의존하므로 특성을 잃은 축소 입력은 다른 분기를 타 재현이 아니게 된다.
- self-test(validate_harness.py)의 정본 트리거 문언은 "harness 데이터·문서 변경"이며 코드 변경은 그 밖이다(CLAUDE.md·memory-harness·관리_절차 4절 공통 문언 확인). 코드 변경 시 실행은 유익하나 "정본 요구"가 아니라 "보수적 확장"으로 표기해야 정확하다. 반증이 잡은 이 문언 드리프트는 규범 인용 시 문언 축자 확인의 필요를 보여 준다.

## 작업 너머 시사점

- M-0026(extract_all.py 파라미터 개선)이 커스텀본의 예시 절로 재사용됐다. 재현 확정, 실패 관찰, 요청받은 파괴적 개선의 실측 기각, 최소 수정, 회귀 확인의 전 절차를 밟은 실제 사례가 가상 예시보다 시연 가치가 높다. 기억 그래프의 M 노드가 스킬 본문의 소재가 된 첫 사례.
- 커스텀본이 원본 대비 명시적으로 얻은 것: 관할 지도(산문은 legal-harness), harness 불변 자산 게이트, 로컬 특유 합리화 반박("스위트가 없으니 불가능하다" 류), stderr 정상 진행 로그의 조건화. 명시적으로 잃은 것(정직하게 기재): 상주 자동 회귀망. mock 프레임워크 도입 시 원본 리소스 재검토가 트리거로 걸려 있다.
