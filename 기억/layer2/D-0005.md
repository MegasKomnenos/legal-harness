## 종합

두 노드는 표면적으로 "PDF 생성 도구 작업"이지만, 관통하는 하나의 메커니즘은 `generate_pdf.py`가 **표현(무엇을 어떤 서식으로 그릴지, HTML/CSS)을 렌더링(어떻게 픽셀화할지, 엔진)에서 분리**한 아키텍처 결정이다. 이 분리가 이식성, 단일 수정점, 검증이라는 세 가지 서로 다른 이득을 동시에 낳는다.

근거 사슬은 이렇다. 스크립트는 `parse()`(txt→구조, L146) → `render()`(구조→HTML/CSS, L257) → `write_pdf()`(변환, L306)의 3단이고, 이 중 `render`가 만드는 HTML/CSS가 파이프라인의 유일한 표현원이다. CSS는 모듈 상수 한 곳(L26-108)에 있고 `render`가 `<style>{CSS}</style>`로 주입한다(L262). 반면 환경 의존성은 전부 마지막 픽셀화 단에 몰려 있다: WeasyPrint의 GTK 런타임 부재(M-0006), poppler 부재로 인한 PDF→이미지 렌더 불가와 MCP file: 프로토콜 차단(M-0009). 정확성의 본체(무엇이 어떤 서식으로 배치되는가)는 표현원에 있고, 환경이 막는 것은 그 아래 렌더링 층이다. 이 비대칭에서 세 결과가 파생된다.

**첫째, 이식성.** 표현원이 렌더러 독립적이므로 엔진만 WeasyPrint→Playwright로 교체해도 서식이 대체로 불변이다. `write_pdf`의 try/except 폴백(L313-328)이 이를 실현하고, Playwright는 `chromium.launch(channel=channel)`(호출은 L351, 채널 리터럴 `'msedge'/'chrome'`은 L349)로 시스템 브라우저를 재사용해 번들 다운로드 없이 가용성을 넓힌다(M-0006). 다만 이 등가는 자동이 아니다. Chromium이 CSS Paged Media의 margin box(`@bottom-center`)를 지원하지 않으므로, 페이지번호 `- N -`은 WeasyPrint 경로에서는 `@bottom-center`(L30-34)로, Playwright 경로에서는 `page.pdf`의 `footer_template`과 `prefer_css_page_size=False`(L341-370)로 각각 재현한다. 즉 폴백 체인은 "동일 표현원 + 렌더러별 차이를 흡수하는 보정 레이어"의 조합이며, 표현원 공유는 서식 일치의 근사이지 항등이 아니다.

**둘째, 단일 수정점.** 표현원이 한 곳(CSS 상수)이라 서식을 고치면 두 엔진 모두에 반영된다(M-0009). 첫줄 들여쓰기 도입(`p { text-indent: 1em }`, L67)이 그 예다. 그런데 이 힘에는 이면이 있다: 한 규칙이 의도보다 넓게 적용될 위험이다. `text-indent`는 상속 속성이고 `.sub`/`.i1`/`.i2`/`.foot p`가 모두 `<p class=...>`로 렌더링되어 `p` 규칙에 매칭되므로, 네 예외에 명시적 `text-indent: 0` override가 필수다(L74, 78-79, 100). 특정성(`.sub` 0,1,0 > `p` 0,0,1)이 override를 보장한다. 표현원의 표현력을 넓힐수록 예외 봉쇄를 명시해야 한다는 짝 관계다.

**셋째, 검증 축의 이동.** 시각(픽셀) 검증이 막힌 환경에서도 정확성의 본체가 표현원에 있으므로, 표현원과 그 텍스트 산물을 검증하면 최종 PDF의 정확성을 상당 부분 담보할 수 있다. 이 환경의 제약은 구체적으로 세 가지다: poppler(pdftoppm) 부재로 PDF를 이미지로 못 열어 Read PDF가 실패하고(M-0009), Playwright MCP는 file: 프로토콜을 차단하며(M-0009), 따라서 시각 스크린샷 경로가 봉쇄된다. 갈음 수단도 세 가지다: `--html` 직접 출력으로 요소-클래스-CSS 대응을 구조 검증하고(M-0009), PyMuPDF(fitz) 텍스트 추출로 페이지 수와 footer를 내용 검증하며(M-0006의 7페이지 `- 1 -`~`- 7 -` 확인), `write_pdf`가 반환해 `main`이 출력하는 엔진명(L402-403)으로 어느 경로로 생성됐는지 확인한다. 세 갈음 모두 "픽셀"이 아니라 "표현·텍스트 층"을 본다는 점에서 위 분리 원리의 직접적 귀결이다.

마지막으로 결이 다른 인접 통찰 하나(M-0009 단독 근거이므로 좁게 유지한다): 표준을 정하는 원본 데이터가 분기할 때(첫줄 들여쓰기 원본 4건 중 사용 2건 대 미사용 2건), 데이터만으로 결정되지 않으므로 사용자 선호로 재결정하고, 이때 판단표의 채택 셀뿐 아니라 근거 셀도 함께 갱신해야 표와 채택이 정합한다. 이는 PDF에 국한되지 않는 "판단표 문서화 정합" 원리로 읽히지만, 근거가 한 건이므로 일반화는 유보한다.

## 근거

- **M-0006** [WeasyPrint→Playwright 폴백 도입]: parse→render→변환 3단으로 엔진 교체가 서식에 영향을 주지 않는 점, 렌더러 차이(@bottom-center 미지원)를 엔진별 보정으로 흡수한 점, poppler 부재를 PyMuPDF 텍스트 추출로 갈음한 점의 원천.
- **M-0009** [첫줄 들여쓰기 도입]: 단일 표현원(CSS 상수)을 두 엔진이 공유해 한 곳 수정이 양쪽에 반영되는 점, `text-indent` 상속·특정성 함정과 4종 override, MCP file: 차단·poppler 부재로 인한 시각 검증 제약과 `--html` 구조 검증 갈음, 원본 분기 시 사용자 선호 재결정과 근거 셀 동기화의 원천.

## 제안

- (승인 대기, harness 소스 변경 수반) 현재 검증은 `--html` 구조 확인, PyMuPDF 텍스트 추출, 엔진명 출력의 세 수단으로 흩어져 있고 매번 수작업으로 조합된다. 이를 하나의 검증 절차 또는 스크립트 옵션(예: 생성 직후 페이지 수·footer 패턴·주요 클래스 존재를 자동 점검하는 `--verify` 류)으로 묶으면, 시각 검증이 막힌 이 환경에서 검증의 재현성과 누락 방지가 올라간다. 근거가 두 노드뿐이므로 도입 여부는 실사용 빈도를 보고 판단할 것을 제안하며, 지금 적용하지는 않는다.
- 폴백 체인에 제3 엔진을 추가할 필요성은 현재 근거에서 도출되지 않는다. 제안하지 않는다.

## 수렴·유연성 원장

- **가정**: 정확성의 본체가 표현원(HTML/CSS)에 있고 환경 의존성은 픽셀화 단에 몰려 있다는 비대칭.
- **강화(소스 대조)**: 코드 대조에서 CSS 단일 상수와 3단 분리, try/except 폴백이 실제 구현되어 있음을 확인했고(L26-108, L306-328), 두 노드가 독립적으로 같은 분리에 기대어 서로 다른 문제(이식성·서식·검증)를 풀었다. [[D-0002]]의 "종합 대상이 프로젝트 소스면 그 소스를 대조한다" 휴리스틱이 이 종합에서도 유효했다.
- **적대 검증이 교정한 것**: 초안의 소스 라인 인용 2건이 부정확했다. `<style>` 주입은 L261이 아니라 L262(L261은 `<html>` head 라인), `chromium.launch` 호출은 L349가 아니라 L351(L349는 채널 리터럴 정의)이었다. 독립 검증 서브에이전트가 소스를 전수 대조해 잡았고 정정했다. 나머지 라인 인용(parse L146, render L257, write_pdf L306, CSS L26-108, 폴백 L313-328, override L74·78-79·100 등)과 근거 도출성·과대일반화·basis는 모두 건전 판정.
- **반증·경계**: "표현원 공유 = 서식 동일"은 근사임이 한 지점에서 드러났다(@bottom-center 미지원을 엔진별 보정으로 메움). 더 취약할 후보로, 시스템에 Noto Serif CJK KR이 없을 때 두 엔진이 서로 다른 폴백 폰트로 렌더하면 자간·줄바꿈이 갈릴 수 있으나 이는 미검증이다.
- **다음에 살필 후보**: (a) 지정 폰트 부재 시 WeasyPrint와 Chromium의 폰트 폴백 동작 차이, (b) PyMuPDF 텍스트 추출이 놓치는 레이아웃 정밀도(페이지 넘김 위치·자간·줄간격)의 범위와 갈음 한계, (c) 근거가 2개뿐이므로 "표현/렌더링 분리" 원리가 다른 harness 산출물 생성 도구에도 성립하는지는 사례가 더 쌓인 뒤 판단.
